# ImaginAI - システム仕様

## 1. システム概要

ImaginAIは、AI画像・動画・音楽生成APIを専用GUIで操作するデスクトップアプリケーションである。Electronで構築され、Google Gemini API（画像生成）、Veo 3.1（動画生成）、Lyria 3（音楽生成）に対応し、将来的に他のAPIプロバイダへの拡張が可能な設計となっている。

## 2. アーキテクチャ

### 2.1 プロセスモデル

- **Mainプロセス**: IPC通信、ファイルI/O、API呼び出し、設定管理、APIキー暗号化、ウィンドウ管理を担当する。
- **Preloadプロセス**: `contextBridge`を通じてMainプロセスとRendererプロセスを安全に橋渡しする。
- **Rendererプロセス**: React + MUI によるUIを担当する。

## 3. 機能仕様

### 3.1 画像生成

- ユーザーはモデルを選択し、パラメータ（アスペクト比、解像度、生成枚数）を設定し、プロンプトを入力し、任意で参照画像を添付する。
- パラメータはモデルの対応状況に基づいて検証される（例: 21:9は対応モデルのみ、4Kは対応モデルのみ）。モデルが画像入力非対応の場合は参照画像UIが非表示、ネガティブプロンプト非対応の場合はネガティブプロンプト欄が非表示、生成枚数が1枚のみのモデルでは枚数スライダーが非表示となる。
- 出力フォーマットはPNG固定。セーフティフィルターはAPI側で制御されており、ユーザー選択不可。
- 参照画像はファイル選択、ドラッグ＆ドロップ、または履歴エントリから追加できる。サムネイルとして表示され、個別に削除可能。
- 「生成」ボタンでAPIリクエストを送信する。生成中はボタンが無効化され、ローディングインジケータが表示される。
- 成功時は結果が自動的に履歴に保存される。失敗時のメッセージは2層構造で扱う:
  - **インフラ系エラー**（HTTP 4xx/5xx、gRPCステータス、ネットワーク等）: ユーザー向けに行動指針付きで言い換えたローカライズ文言を表示。
  - **コンテンツ系ソフト失敗**（HTTP 200だが生成物0件: Safetyブロック、Recitation、モデル拒否等）: 「{種別}が生成されませんでした。詳細を確認してください。」のみ事実として表示する（プロンプト変更を促す不確かなアドバイスは出さない）。
- 「詳細」リンクをクリックすると、APIから返された診断情報（`finishReason`、`promptFeedback.blockReason`、`safetyRatings`、テキストパートに含まれる拒否理由、Veoの`raiMediaFilteredReasons`等）を生で確認できる。
- ネットワークエラー発生時はリトライボタンを表示し、ユーザーがリトライ可否を選択できる。
- 生成前に履歴保存先のディスク空き容量を確認し、100MB未満の場合は警告を表示する。
- 履歴件数が設定された上限（10,000件）を超えている場合、生成がブロックされる。

### 3.2 履歴管理

- 各生成結果はメタデータ（JSON）と生成ファイル（画像/動画/音楽）として保存される。
- 履歴はサムネイルグリッドでフレックスレイアウト表示され、更新日降順でソートされる。ページネーション（50件ずつ）によるインフィニティスクロール（遅延ロード）で表示される。
- 左クリックでモードレスビューアウィンドウに画像を表示する（動画の場合は動画ビューアで再生、音楽の場合は音楽プレイヤーで再生）。複数枚生成の場合、すべてをそれぞれ別ウィンドウで開く。
- 右クリックでコンテキストメニューを表示: プロンプトに追加、名前をつけて保存（複数枚の場合は各画像のダイアログを順次表示）、パラメータ復元、削除。
- ヘッダーメニュー: すべて削除、すべて圧縮保存して削除（ZIPエクスポート、圧縮中はプログレスバーを表示）。
- 各エントリには生成所要時間（API呼び出し開始から完了まで）が記録され、履歴カードに表示される。
- 履歴パネルの高さは上辺のドラッグで調整可能。
- プロンプトテキストによる部分一致検索でフィルタリング可能。

### 3.3 動画生成

- Veo 3.1モデル（Lite/Fast/Standard）による動画生成に対応する。
- パラメータ: 長さ（4/6/8秒）、動画解像度（720p/1080p/4k）、アスペクト比（16:9/9:16のみ）、ネガティブプロンプト、シード値。
- Veoモデル選択時は画像固有のUI（出力解像度）が非表示となり、動画固有のUI（長さ、動画解像度、シード値）が表示される。
- 動画生成はVeo APIの非同期ポーリング（Long-Running Operation）方式で行われ、生成中は経過時間が表示される。ポーリング間隔: 10秒。
- 生成された動画はMP4形式で履歴ディレクトリに保存される。
- サムネイルはffmpegにより1秒目のフレームから抽出される。ffmpegが利用不可の場合はサムネイルなし。
- 1080p/4K解像度の場合、長さは8秒のみ指定可能（生成ボタンで検証）。
- 参照画像を添付することで画像→動画生成（Image-to-Video）が可能。
- 音声はモデルにより自動的に生成される（ユーザー制御不可）。
- 人物生成の制御はモードに応じてAPIが自動的に適用する（ユーザー制御不可）。

### 3.4 音楽生成

- Lyria 3モデル（Clip/Pro）による音楽生成に対応する。
- **Lyria 3 Clip** (`lyria-3-clip-preview`): 30秒の短いクリップを生成。$0.04/曲。
- **Lyria 3 Pro** (`lyria-3-pro-preview`): 最大約3分のフルレングス楽曲を生成。$0.08/曲。曲の長さはプロンプトで指定。
- Lyriaモデル選択時は画像・動画固有のUI（アスペクト比、解像度、生成枚数、長さ、シード値）がすべて非表示となる。
- プロンプトでジャンル、雰囲気、楽器、BPM、キー、歌詞（`[Verse]`, `[Chorus]`, `[Bridge]`セクションタグ）、タイムスタンプ指定が可能。歌詞に関する指示をプロンプトに含めることで、歌詞付き楽曲を生成できる。
- 参照画像を添付することで画像→音楽生成（Image-to-Music）が可能。Lyria の公式上限は 10 枚だが、本アプリでは `ModelDefinition.maxReferenceImages` で **5 枚に制限**している。

#### 参照画像の前処理（image / video / music 共通）

全ての生成経路で、APIに送る前に Electron `nativeImage` で以下の前処理を行う:

1. デコード
2. 長辺が **1920px** を超える場合のみ、縦横比を維持して縮小（`quality: 'best'`）
3. **JPEG (品質 85) に再エンコード**（PNG/WebP 入力も JPEG に統一）

枚数上限は各モデル定義の `maxReferenceImages` フィールドで宣言する（省略または 0 は画像入力非対応）。現状の値:

| モデル | maxReferenceImages |
| --- | --- |
| Nano Banana 系 (gemini-*-image*) | 10 |
| Imagen 4 系 | 未定義（画像入力非対応） |
| Veo 3.1 系 | 1 |
| Lyria 3 系 | 5 |
| Gemini TTS 系 | 未定義（画像入力非対応） |

動画 (Veo) は枚数 1 の単一スロットモデルとして扱い、UI で新しい画像を添付すると **直前の画像を上書き**する。参照画像エリアには「開始フレーム」ラベルを表示してユーザに用途を明示する。

- ネガティブプロンプトは非対応。
- 生成された音楽は履歴ディレクトリに保存される。保存形式（拡張子）はAPIが返した`mimeType`に基づいて決まる（通常はMP3、48kHz ステレオ、SynthIDウォーターマーク付き）。
- APIレスポンスにテキストパート（歌詞、楽曲説明、BPM等のメタデータ）が含まれる場合、`audioTexts`フィールドにparts順序のまま配列として保存される。テキストパートの内容や数はAPIの挙動に依存し、構造的なラベル区分はない。
- 履歴パネルではサムネイルの代わりに音楽アイコンで表示される。
- クリックで音楽プレイヤーウィンドウ（モードレス）が開き、HTML5 `<audio>` 要素による再生コントロール付きで自動再生される。テキストパートがある場合はプレイヤーウィンドウ内にパートごとに区切り線付きで表示される。
- APIは`/v1beta/models/{model}:generateContent`エンドポイントを使用し、`responseModalities: ['AUDIO', 'TEXT']`を指定する。レスポンスのpartsの順序は保証されないため、すべてのpartsをイテレートして音声データ（`inlineData` with `audio/*`）とテキストデータを分離する。
- Single-turn only（マルチターン非対応）。

### 3.4.1 音声合成（TTS）

- Gemini TTSモデル（Flash TTS Preview / Pro Preview TTS / Flash Preview TTS）によるテキスト読み上げ生成に対応する。
- **Gemini 3.1 Flash TTS Preview** (`gemini-3.1-flash-tts-preview`): 最新モデル。Audio Tags対応。無料枠 1,500回/日。
- **Gemini 2.5 Pro Preview TTS** (`gemini-2.5-pro-preview-tts`): 品質重視。無料枠 50回/日。
- **Gemini 2.5 Flash Preview TTS** (`gemini-2.5-flash-preview-tts`): コスパ重視。無料枠 2,000回/日。
- TTSモデル選択時、ParameterPanelに以下を追加表示する:
  1. **スタイル**: 9種のプリセット + `カスタム`の選択肢を持つドロップダウン（表示形式 "スタイル名 : 効果"）。プリセット選択で下段のスタイル内容欄に該当する英語プロンプトが自動展開される。
  2. **スタイル内容**: 複数行テキスト入力。手動で編集するとスタイル選択は自動的に`カスタム`になる。
  3. **ボイス**: 30種の事前定義ボイスを "ボイス名 : 特徴" 形式で選択。
- プロンプト欄のラベルは「話させたい内容」に差し替わる。参照画像UIは非表示。
- Gemini 3.1 Flash TTS Preview のみ、プロンプト欄ラベル右に `?` アイコンを表示。クリックで Audio Tags 詳細解説ダイアログ（Expressions / Prosody / Pause セクション）を開く。
- APIリクエスト:
  - エンドポイント: `/v1beta/models/{model}:generateContent`
  - `generationConfig.responseModalities: ['AUDIO']`
  - `generationConfig.speechConfig.voiceConfig.prebuiltVoiceConfig.voiceName` に選択したボイス名を指定
  - テキスト本文は `Style: <英語スタイル指示>, Text: <ユーザー入力>` の形式で合成（スタイル指示が空なら本文のみ）
- APIは raw PCM（例: `audio/L16;codec=pcm;rate=24000`）で返却される。後処理は次の順で行う:
  1. メモリ上で同梱ffmpegを介して MP3 に変換（CBR 160 kbps, 48 kHz, mono）。成功時は `.mp3` で保存。
  2. ffmpegが利用不可・失敗した場合は、純JSで RIFF/WAV ヘッダ（44バイト）を付与した WAV を `.wav` で保存。これによりAPIが成功している（=トークンが消費されている）にもかかわらず後処理失敗で結果が破棄されることを防ぐ。
- `HistoryEntry.styleInstruction` / `voice` にも保存され、履歴からの復元で元のスタイル/ボイスに戻る。
- 1リクエストあたりの最大テキスト長は 4,000 バイト（API制限）。

### 3.5 画像ビューア

- モードレスウィンドウとして開く（複数同時に開くことが可能）。
- 初期サイズは画像の等倍ピクセルサイズ。画面サイズの90%を超える場合は縮小して表示する。
- ウィンドウリサイズ時に画像は縦横比を維持して拡大・縮小する。

### 3.6 動画ビューア

- モードレスウィンドウとして開く。HTML5 `<video>` 要素による再生コントロール（再生/一時停止、シーク、音量）付き。
- 自動再生で開始される。

### 3.7 設定

- **言語**: i18nextによる日本語/英語切り替え。変更は即時反映。翻訳データは`src/renderer/i18n/locales/`に一元管理される。
- **テーマ**: ライト/ダーク/システム（OS追従）。変更は即時反映。
- **APIキー**: Electronの`safeStorage`で暗号化保存。マスク表示と表示/非表示トグル付き。以下の3種類のキーを最大7つ（デフォルト1・無料枠1・カスタム最大5）まで保持可能。
  - **デフォルトキー**: タイトルなし。通常利用想定。
  - **無料枠キー**: タイトルなし。このキーを選択中はモデルの料金ではなく無料利用枠の情報が表示される。
  - **カスタムキー**: タイトル付きで最大5つ追加可能。各キーに「無料枠キーとして扱う」フラグを設定できる。
  - キャプションバー（タイトルバー）の切替UIで、現在有効なキーを切り替える。アクティブキーが生成APIリクエストに使用される。
  - アクティブキーが無料枠種別の場合、ParameterPanelではモデル毎の参考価格は非表示となり、代わりに無料利用枠の情報（利用可否等）が表示される。両者は相反関係で同時表示されない。
- **履歴保存先**: ユーザーが任意のディレクトリを選択可能。変更時に既存履歴の移動有無を確認するダイアログを表示。

## 4. データフロー

### 4.1 生成フロー

1. Renderer: ユーザーがパラメータを設定し、生成ボタンをクリック
2. Renderer -> Main (IPC): `generation:execute`でGenerationParamsを送信
3. Main: 履歴件数が上限を超えていないか検証
4. Main: API呼び出し
   - 画像: Imagen predictまたはGemini generateContent
   - 動画: Veo predictLongRunning → LROポーリング → 動画ダウンロード
   - 音楽: Lyria generateContent → 音声・テキストパート抽出
5. Main: 生成ファイルとメタデータを履歴ディレクトリに保存
6. Main: サムネイルを生成（画像: Electron nativeImage、動画: ffmpeg、音楽: サムネイルなし）
7. Main -> Renderer (IPC): HistoryEntryを返却
8. Renderer: 履歴一覧を再読み込み

### 4.2 API通信

- **Imagenモデル**（例: imagen-4.0-generate-001）: `/v1beta/models/{model}:predict`エンドポイントを使用。ネガティブプロンプトはテキストプロンプト内に埋め込み。`sampleCount`で複数枚生成対応（1-4枚）。
- **Nano Bananaモデル**（例: gemini-3.1-flash-image-preview）: `/v1beta/models/{model}:generateContent`エンドポイントを使用し、`responseModalities: ['TEXT', 'IMAGE']`を指定。ネガティブプロンプトはテキストプロンプト内に埋め込み。1枚ずつ生成。参照画像入力対応。
- **Veo 3.1モデル**（例: veo-3.1-generate-preview）: `/v1beta/models/{model}:predictLongRunning`エンドポイントを使用。レスポンスはLong-Running Operation（LRO）として返され、`/v1beta/{operationName}`を10秒間隔でポーリングする。完了時に動画のダウンロードURLを取得し、`x-goog-api-key`ヘッダーで認証してダウンロードする。ネイティブ音声生成を含む。ポーリングはタイムアウトなしで完了まで待機する（公式SDK準拠）。
- **Lyria 3モデル**（例: lyria-3-pro-preview）: `/v1beta/models/{model}:generateContent`エンドポイントを使用し、`responseModalities: ['AUDIO', 'TEXT']`を指定。レスポンスのpartsにはaudio（inlineData, audio/mpeg）とtext（歌詞・楽曲説明・メタデータ等）が混在し、順序不定のためイテレートして分離する。テキストpartsは構造的なラベル区分がないため、配列としてそのまま保存する。参照画像入力対応（Image-to-Music）。
- 参照画像はリクエスト時にローカルファイルパスからBase64エンコードして送信
- APIキーはリクエスト時に暗号化ストレージから取得

## 5. セキュリティ

- APIキーはElectronの`safeStorage`により暗号化して保存される。
- RendererプロセスはAPIキーに直接アクセスせず、すべてのAPI操作はMainプロセス経由のIPC通信で処理される。
- コンテキスト分離とPreloadスクリプトにより、WebコンテンツとNode.js APIの間に安全な境界を確保する。

## 6. 拡張性

- モデル定義は`shared/constants.ts`に一元管理されている。新モデル追加時はこのファイルのみを更新する。
- 各モデル定義には対応パラメータが含まれ、UIの有効/無効制御が自動的に行われる。生成バックエンドのディスパッチは`mediaType`で行い、画像モデル内のImagen/Gemini区別のみモデルID接頭辞（`imagen-*`はImagen API）で判別する。ネガティブプロンプトのAPI対応可否（`apiNegativePrompt`）もモデル定義で管理される。
- モデル定義のAPIプロバイダフィールドにより、将来的な複数プロバイダ対応に備えている。
- APIキーストレージはプロバイダ別にキー管理されており、複数APIキーに対応可能。
